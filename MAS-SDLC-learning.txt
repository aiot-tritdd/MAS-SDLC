os.getenv() — how it works

  DEBUG = os.getenv('DEBUG', 'False') == 'True'

  3 things happening:

  1. os.getenv('DEBUG', 'False') — read DEBUG from .env. If it doesn't exist, use 'False' as
  default.
  2. Everything in .env is a string — even True comes out as "True", not a real boolean.
  3. == 'True' — convert the string to a real boolean. "True" == 'True' → True. "False" ==
  'True' → False.

  Same pattern for SECRET_KEY:
  SECRET_KEY = os.getenv('SECRET_KEY')
  No default needed — just read the value from .env.

PATH — what it is and why it matters

  ---
  What is PATH?

  PATH is an environment variable — a list of folders your Mac searches when you type a
  command in terminal.

  When you type psql:
  1. Mac looks through each folder in PATH one by one
  2. If it finds psql in any folder → runs it
  3. If it finds nothing → "command not found"

  ---
  View your current PATH:
  echo $PATH
  Folders are separated by :.

  ---
  Why psql doesn't work:

  PostgreSQL lives in /usr/local/opt/postgresql@16/bin/ — that folder is NOT in your PATH.
  Mac can't find it.

  ---
  How to add it permanently:
  echo 'export PATH="/usr/local/opt/postgresql@16/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc

  - ~/.zshrc = config file that runs every time you open a terminal
  - export PATH="new_folder:$PATH" = prepend the new folder to the existing list
  - source ~/.zshrc = reload the file right now without reopening terminal

  ---
  After this: typing psql just works. No more full paths.

migrate — what it does and why

  ---
  What is a migration?

  A migration = a blueprint for changing the database. It describes: "create this table, add
  this column, change this field."

  Django stores migrations as Python files in each app's migrations/ folder.

  ---
  What does migrate do?

  1. Looks at all migration files across all installed apps
  2. Checks which ones have already been applied (tracks this in a django_migrations table)
  3. Runs any that haven't been applied yet — creates/updates tables in the database
  4. Each OK = one migration applied successfully

  ---
  The two commands you'll always use together:

  python manage.py makemigrations   # Step 1: generate migration file from your models
  python manage.py migrate          # Step 2: apply migration files to the database

  - makemigrations = "look at my models, generate the blueprint"
  - migrate = "take the blueprint, build the actual tables"

  ---
  What just ran:

  Django's built-in apps (auth, admin, sessions, contenttypes) all had their tables created
  in our PostgreSQL database for the first time.

  When WE write Ticket, AgentRun, Artifact models — we'll run both commands: makemigrations
  first, then migrate.

Step 1.4 — Configure Root URLs

  ---
  Two files touched:

  File 1 — sdlc_spike/urls.py (the ROOT router)

  from django.contrib import admin
  from django.urls import path, include

  urlpatterns = [
      path('admin/', admin.site.urls),
      path('api/', include('core.urls')),
  ]

  This is Django's front door. Every incoming request hits this file first.
  - path('admin/', ...) → Django's built-in admin panel
  - path('api/', include('core.urls')) → anything starting with /api/ gets
    forwarded to core/urls.py to handle

  include('core.urls') means: "go look inside the core app's urls.py file
  for more specific routes." Django splits routing into two levels:
  - Root urls.py = top-level routing (which app handles this prefix?)
  - core/urls.py = detail routing (which view handles this specific endpoint?)
